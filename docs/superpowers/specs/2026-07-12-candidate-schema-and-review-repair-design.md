# 候选数据契约与审核页修复设计

## 背景

真实 DeepSeek 冒烟运行把 `demand_evidence` 与 `risks` 返回为字符串。候选生成流程只检查字段是否存在，没有检查字段类型；Jinja 模板随后把字符串按字符遍历，造成“一字一个圆点”和“逐字分号”。同类弱校验也存在于来源包、平台文案与包装数据中，继续选择候选可能把错误传递到正文和图片阶段。

## 目标

1. 模型输出不符合约定时立即、清晰地失败，不生成可选择的错误候选。
2. 候选页在桌面端和窄屏下正确呈现，并为提交失败提供可恢复状态。
3. 后续来源包、正文、平台稿和包装数据使用同一套严格类型校验。
4. 废弃当前仅基于冒烟测试资料得到的 88 分候选；修复后只用可追溯需求材料重新生成。

## 方案比较

### 方案 A：模板兼容错误类型

在模板中检测字符串并把它包装成单元素列表。改动最少，但会掩盖模型输出错误，后续生产仍可能带病运行，不采用。

### 方案 B：候选阶段严格校验

只修候选数据契约和页面。能解决当前截图问题，但正文、包装与卡片阶段仍存在同类风险，不采用。

### 方案 C：端到端严格数据契约（采用）

在候选与生产各阶段进行明确的字典、字符串、字符串数组和嵌套对象校验；提示词同时声明精确 JSON 类型；页面只消费已经验证的数据。该方案改动集中、没有新增依赖，也能阻止错误继续传播。

## 设计

### 数据契约

- 顶层模型响应必须是字典。
- `candidates` 必须是数组，每个候选必须是字典。
- 候选 `id`、`title`、`category`、`trigger`、`audience`、`differentiation` 必须是非空字符串。
- `demand_evidence`、`risks`、`source_ids` 必须是非空或允许为空的字符串数组；其中需求证据和来源 ID 不得为空。
- 候选 ID 必须唯一，评分字段必须精确匹配既定六项整数范围。
- 来源包、母稿、平台稿、包装与小红书卡片分别验证必需字段、字段类型、精确数量及嵌套结构。

校验失败时抛出包含阶段和字段路径的 `ValueError`，不把错误数据写入候选文件或任务目录。

### 审核页

- 保留现有视觉风格，不重新设计品牌。
- 列表字段只按完整条目渲染。
- 移除内联 `onclick`，改用 `data-candidate-id` 与统一事件监听。
- 提交时禁用全部候选按钮，成功后刷新；失败时恢复按钮并显示可读错误。
- 状态提示增加 `role="status"`、`aria-live="polite"`，按钮增加明确焦点样式。
- 空候选批次显示说明，不渲染空白页面。

### 真实需求门槛

当前 `wiki/smoke-test.md` 只用于连通性测试，不能作为真实选题证据。现有 2026-07-12 候选批次标记为无效并退出审核流程。修复验证使用合成测试数据；重新生成真实候选前，必须提供带路径或 URL 的可追溯需求材料。

### 测试策略

按测试驱动方式执行：

1. 先增加字符串冒充数组的回归测试并确认失败。
2. 增加候选顶层、字段类型、候选 ID、评分结构测试。
3. 增加生产各阶段错误类型测试，确保在写任务文件前失败。
4. 增加审核页完整条目、无内联脚本、失败恢复与空状态测试。
5. 运行全量单元测试，并在本地浏览器进行桌面和窄屏视觉复查。

## 不在本次范围

- 不接入新的模型供应商。
- 不改变公众号或小红书内容策略。
- 不自动发布内容。
- 不把冒烟测试数据包装成真实市场证据。

## 完成标准

- 截图中的逐字列表问题有自动化回归测试覆盖。
- 任一模型阶段返回错误类型时，流程在消费或落盘前停止并给出字段级错误。
- 候选页提交失败后可再次操作，键盘焦点清晰，空状态可理解。
- 全量测试通过，修复后的页面经真实浏览器截图复核。
- 旧的 88 分冒烟候选不再可被选择。
