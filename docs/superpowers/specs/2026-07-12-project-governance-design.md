# 项目治理与上下文防遗忘设计

## 目标

让任何一次新的 Codex 会话都能在不依赖长对话记忆的前提下，恢复项目目标、当前状态、内容边界和下一步，并在偏离规则时停止而不是猜测。

## 权威层级

发生冲突时按以下顺序处理：

1. 用户当前明确指令；
2. 项目根目录 `AGENTS.md`；
3. 本地 `content-ops/state/project-state.yaml`；
4. `content-ops/config/editorial-policy.yaml`；
5. `docs/DECISIONS.md` 与设计文档；
6. 历史对话。

Codex 不得静默化解冲突。若更高层指令会改变长期定位、外部发布、成本或安全边界，必须先说明影响。

## 文件职责

- `AGENTS.md`：稳定且必须遵守的项目宪法、开工与收工仪式。
- `content-ops/config/editorial-policy.yaml`：机器可读的选题、证据、内容比例、停止条件与人工闸门。
- `content-ops/config/project-state.example.yaml`：可公开的状态结构示例。
- `content-ops/state/project-state.yaml`：本地真实状态，继续保持 Git 忽略，不进入公开仓库。
- `docs/DECISIONS.md`：长期决策、理由与允许重议的条件。
- `docs/content-release-checklist.md`：人工发布前检查项。
- `docs/content-ops-runbook.md`：运行与故障恢复命令。

## 开工仪式

Codex 开始任何任务前必须：

1. 读取 `AGENTS.md`；
2. 读取本地状态文件，缺失时只允许用示例初始化，不得假设账号或任务状态；
3. 检查 Git、候选、草稿、审核页、端口和计划任务状态；
4. 说明当前阶段、阻塞点和本次动作；
5. 再执行写操作。

## 收工仪式

完成任务前必须：

1. 运行与风险相称的测试和页面检查；
2. 更新本地状态、下一步、阻塞点和最近验证；
3. 战略变化写入决策记录；
4. 报告真实完成项、未完成项和需要用户做的事；
5. 不把待验证事项写成成功。

## 内容与商业证据

- 无可追溯需求证据，不进入正式候选。
- 浏览、点赞、收藏、评论、关注、咨询、询价、预付款、付款与复购分层记录，低等级证据不能自动升级。
- 冷启动内容结构为相邻泛流量 60%、真实实验/工作流 30%、长期深度方向 10%。
- 单篇内容不能改变长期定位；至少观察五篇同类内容或连续两周数据后再调整。
- 没有合格选题时允许当日不生产。

## 人工与自动化边界

人工必须完成选题确认、终稿审核和平台发布。系统不得保存平台密码、Cookie、OTP 或 Token，不得自动登录或发布，不得在结构校验、来源校验或质量检查失败后继续生成。

## 公开与私有边界

公开仓库只保存通用规则、示例状态和代码。真实账号状态、私人来源、生成稿件、评论/私信、API Key 与本机绝对路径只保存在被 Git 忽略的本地状态或个人知识库中。

## 验证

新增治理测试负责确认规则文件存在、YAML 可解析、关键阈值和人工闸门未被意外删除。现有全量测试继续验证内容生产流程。
