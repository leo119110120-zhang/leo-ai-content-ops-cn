# Leo AI Content Ops · Codex 项目规则

本文件是本仓库的长期项目宪法。聊天记录可以提供背景，但不能代替这里的规则、当前状态和可验证证据。

## 1. 权威顺序

发生冲突时按以下顺序处理：

1. 用户当前明确指令；
2. 本文件；
3. `content-ops/state/project-state.yaml`；
4. `content-ops/config/editorial-policy.yaml`；
5. `docs/DECISIONS.md`、设计文档和运行手册；
6. 历史对话。

Codex 必须报告冲突，不得为了继续执行而静默选择较方便的解释。涉及长期定位、外部发布、付费、删除、隐私或安全边界的变化，必须先说明影响并取得明确授权。

## 2. 每次开工前

Codex 在任何写操作前必须：

1. 完整读取本文件；
2. 读取 `content-ops/state/project-state.yaml`；若不存在，只能读取 `content-ops/config/project-state.example.yaml` 并报告状态未知；
3. 读取 `content-ops/config/editorial-policy.yaml` 和相关决策记录；
4. 检查 Git 状态、待选候选、待审草稿、监听端口和计划任务；
5. 说明当前阶段、阻塞点、本次目标和用户需要参与的环节；
6. 再开始修改或运行。

不得仅凭历史对话推断账号已注册、内容已发布、用户已付款、计划任务已安装或某项测试已通过。

## 3. 原始资料与隐私

- 不得修改 `raw/` 中已有的任何原始文件；程序和 Codex 只能读取新输入。
- 真实账号、评论、私信、客户资料、生成稿件和本地状态不得提交到公开仓库。
- 不得保存或输出平台密码、Cookie、OTP、API Key、支付信息或私人身份信息。
- API Key 只允许从环境变量读取；检查时只能报告“存在/不存在”。
- 对外内容中的截图、手机号、订单号、头像和聊天记录必须先脱敏。

## 4. 需求与证据

- 没有可追溯来源，不得进入正式候选。
- 每个候选必须写清：谁有问题、去哪里解决、已有方案、未解决点、可接受交付、竞品、触达渠道、已知事实和未知项。
- 浏览、点赞、收藏、评论、关注、咨询、询价、预付款、付款、复购和转介绍必须分层记录。
- 低等级证据不得自动解释成高等级证据；评论和领取不能证明付款。
- 没有合格选题时允许当天不生产，不得为了凑数量降低门槛。
- 单篇内容不得改变长期定位；至少观察五篇同类内容或连续两周数据后再调整。

## 5. 内容策略

- 冷启动前 30 篇按相邻泛流量 60%、真实实验/工作流 30%、长期深度方向 10%滚动检查。
- 泛流量必须能回到 AI 实践、真实生意或一人公司，不追无关娱乐热点。
- 公众号与小红书共享事实和核心观点，但不得机械复制。
- 标题承诺必须在正文前 30%兑现；不得使用来源无法支持的夸张结果。
- 不把 AI 生成内容当作事实来源，不虚构用户、数据、案例、成交或平台表现。

## 6. 自动化与人工边界

- 系统最多给出三个合格候选，结构或来源校验失败时立即停止。
- 模型最多重试两次，不得无限循环消耗 Token。
- 当天没有新资料时立即退出，不重复扫描。
- 人工闸门固定为：选题确认、终稿审核、平台发布。
- 不得自动发布公众号或小红书，不得自动登录平台。
- 未经人工选题不得生成正文和图片；未经人工终审不得导出正式发布包。
- 连续两次修改仍不合格时进入人工处理队列，不再自动重写。

## 7. 代码和配置变更

- 修复缺陷或修改行为时先写能复现问题的失败测试，再实现最小修复。
- 战略规则变化先更新 `editorial-policy.yaml` 或 `DECISIONS.md`，再修改代码。
- 保留用户已有改动，不得使用破坏性 Git 命令清除未知内容。
- 公开提交不得包含真实输入、运行状态、绝对本机路径或密钥。
- 完成前必须验证：运行相关测试、检查 `git diff --check`，界面问题还要用真实浏览器复查。

## 8. 每次收工前

Codex 必须：

1. 更新本地 `content-ops/state/project-state.yaml` 的阶段、阻塞点、下一步和最近验证；
2. 将长期决策写入 `docs/DECISIONS.md`；
3. 报告实际完成项、未完成项、风险和用户下一步；
4. 不把待验证事项描述成成功；
5. 保持原始资料、密钥和私人状态不进入 Git。

## 9. 发布完成定义

一篇内容只有同时满足以下条件才算完成：来源可追溯、候选过门槛、双平台适配完成、自动质检通过、人工终审通过、人工上传发布、已安排 24 小时/72 小时/7 天数据回收。
